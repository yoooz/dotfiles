# Craft Review

書いている最中のコードを **「名前」と「分割」の2軸だけ** で深くレビューする軽量スキル。
正しさ（バグ）や簡略化全般は対象外。サブエージェントは使わず、Claude 本体がその場で見る。

## いつ使うか

- ユーザーが「命名見て」「関数分割どう？」「この設計微妙？」「craft-review」と言った
- コーディングが一区切りし、コミット前に名前と構造を整えたいとき

逆に使わない場面（他スキルに委ねる）:

- バグ・正しさのレビュー → `code-review`
- 簡略化・冗長除去全般 → `simplify`
- PR 全体を多観点で徹底レビュー → `my-review`（PR URL を渡されたとき）

## 動作フロー

### Step 1: レビュー対象の確定

- **引数なし**: working tree の変更を対象にする。

  ```bash
  git diff HEAD            # staged + unstaged（追跡ファイル）の変更
  git status --short       # 未追跡ファイルの有無を確認
  ```

  `git diff HEAD` が空なら「レビュー対象の変更がない」と報告して終了。
  未追跡の新規ファイルがあれば、その旨を伝えてレビュー対象に含めるか確認する。

- **引数でファイル/ディレクトリ/関数が指定された場合**: それを対象にする。
  - ファイル全体をレビューしたい意図なら `Read` で通読する。
  - 「変更分だけ」の意図なら `git diff HEAD -- <paths>` を使う。

### Step 2: 文脈を読む（重要）

**diff の hunk だけを見て命名・分割を論じない。** 変更された関数・型については、
必ず周辺を読んでから判断する:

1. diff で変更された関数・型・変数を特定する
2. 該当する関数の**全体**を `Read` で通読する（hunk の外も含めて責務を把握）
3. **命名変更を提案しそうなシンボル**は、`git grep <name>` で呼び出し元・参照箇所を数え、
   リネームの波及範囲（何箇所に影響するか）を把握する
4. プロジェクトの既存コードの命名規約（スネーク/キャメル、接頭辞の慣習など）を周辺から拾う

### Step 3: 2軸でレビュー

#### 軸1: 命名

- **名が実装と乖離**: 関数名と実際の処理がずれている（`save` が実際は upsert、など）
- **抽象度の不一致**: `getData` `handleStuff` `processItem` のように内容を表さない汎用名
- **嘘をつく名前**: `isValid`（bool を返す体）なのに副作用がある／`get` なのに高コスト・I/O を伴う
- **単位・型が名前から読めない**: `timeout` → `timeoutMs`、`size` → `sizeBytes`、`user` が ID か object か不明
- **否定形の bool**: `notDisabled` `isNotEmpty` など二重否定を生む名
- **一貫性の欠如**: 同じ概念に別名（`fetchX`/`getX`/`loadX` が混在）／別概念に同名
- **スコープと名の長さの不釣り合い**: 広いスコープ・長寿命ほど説明的に、ループ内の一時変数は短くてよい
- **規約違反**: プロジェクトの既存命名規約から外れている
- **ニュアンスでより良い候補がある（積極的に提案する）**: 現状の名が「間違い」ではなくても、
  対象の意味・意図・含意をより的確に表す候補が思い浮かぶなら、遠慮せず提案する。
  ただし**必ず複数候補を挙げ、それぞれのニュアンス差を1行で説明**し、現状名と何が違うかを示す。
  例: `delete` vs `remove`（破棄か取り外しか）／`fetch` vs `load` vs `get`（I/O有無・コスト）／
  `check` vs `validate` vs `ensure`（判定か検証か保証か）／`update` vs `sync` vs `apply`。
  これらは severity を **nit**（明確に読みやすくなるなら minor）に置き、「問題」ではなく
  **「より良い候補」**として出す。本当に違いが出ない言い換えなら出さない。

#### 軸2: 関数分割

- **複数責務**: 1関数が「取得 + 加工 + 保存 + 通知」のように複数のことをしている
- **抽象度の混在**: 高レベルの手順と低レベルの実装詳細が同じ関数に同居している
- **クエリとコマンドの混在（CQS）**: 値を返しつつ副作用も持つ。分けると名前も付けやすくなる
- **引数過多**: 4個以上は構造体/オブジェクトにまとめる、または責務過多のサイン
- **フラグ引数**: `doX(flag: boolean)` で内部分岐 → 2つの関数に分けるサイン
- **深いネスト**: ガード節・早期 return で平坦化できる 3段以上のネスト
- **名前を付けにくい関数 = 責務が曖昧なサイン**: 良い名が浮かばないのは、分割すべき兆候として扱う
- **過剰/過少な抽出**: 1回しか使われず可読性も上がらないヘルパー（過剰）／逆に長大な無名処理（過少）

### Step 4: 出力

指摘のみ。**自動修正はしない**（ユーザーが「指摘のみ」を選択）。各 finding は以下の形式:

```
### [軸] severity  file:line — 一言サマリ

**現状**: 何が問題か（1-2行）

Before:
\`\`\`<lang>
<該当コード断片>
\`\`\`

After:
\`\`\`<lang>
<改善案の断片>
\`\`\`

**理由**: なぜ良くなるか（1行）
**波及**: （命名変更の場合のみ）git grep で確認した参照箇所数と、影響ファイルの例
```

**ニュアンス改善の「より良い候補」**を出すときは、After を単一案にせず候補を複数並べ、
それぞれのニュアンス差を添える:

```
### [命名] nit  file:line — `current` のニュアンスを詰められる

**候補**:
- `candidateA` — このニュアンス（現状との違い）
- `candidateB` — このニュアンス（現状との違い）

**波及**: 参照 N 箇所
```

最後に件数を `命名: N件 / 分割: M件` のように要約する。指摘がゼロなら「名前も分割も良好」と伝える。

## severity の目安

- **major**: 名が実装と明確に矛盾し誤読を招く／1関数に明確に3つ以上の責務がある
- **minor**: 読みやすさが目に見えて上がる改善
- **nit**: 好みレベル（より良い候補がある程度）
- **critical は出さない**: 命名・分割は正しさの問題ではないため

## 判断基準（false positive を避ける）

- 改善案で**可読性がむしろ落ちる**なら指摘しない（するなら理由を明記）
- 改善案で**特定のエラーハンドリングや挙動が失われる**なら明記する
- プロジェクトの既存規約・lint に反する提案はしない
- 命名変更は呼び出し元に波及する。**波及範囲が広い割に改善が小さいもの**は severity を下げるか、波及コストを添えて判断材料にする
- 確信が持てない指摘は、前提条件を添えて「要検討」として出す
- ニュアンス改善の「より良い候補」は**積極的に出してよい**が、必ず nit（明確に良くなるなら minor）
  に留め、major/minor の実質的な指摘を埋もれさせない。出力では候補を要約の最後にまとめてもよい

## Out of scope

- バグ・正しさの検証（→ `code-review`）
- 簡略化・冗長除去・パフォーマンス全般（→ `simplify`）
- PR URL を起点にした多観点レビュー（→ `my-review`）
- 修正の自動適用（このスキルは指摘のみ）
- 過去コミットやブランチ間 diff のレビュー（working tree と指定ファイルのみ）
