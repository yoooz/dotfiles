# read-article

記事 URL を「取得 → 要約」するスキル。要約を返すところまでが責務。

## 設計の肝（なぜこの形か）

**要約という判断は Claude 本体（Opus）が本文を読んで行い、スクリプトには決定的な処理だけをさせる。**

過去、`WebFetch` の小型モデルに要約を任せたところ、記事中の「11日/12日/20日」を
「11月/12月/20月」と読み替えるなど、日付や数字を捏造する事故が起きた。
要約を弱いモデルに委譲すると同種の事故が再発するため、本スキルでは
`scripts/fetch_article.py` で**本文テキストだけ**を取得し、要約は本体が行う。

このスキルは**取得と要約だけに責務を絞る独立スキル**。要約結果の記録（デイリーノートへの
つぶやき等）は行わない。記録したい場合は、利用者が別途 `jot` などの記録手段を使う。

## いつ使うか

ユーザーが記事 URL を貼って、次のように言ったとき:

- 「この記事まとめて / 要約して / 読んで / 内容教えて」
- クリッピングしたい記事の中身を知りたいと言われたとき

逆に使わない場面:

- HN のトップ一覧を眺めたい → `hn-reader` スキル
- ファイルやコードの要約（Web 記事ではない）

## 動作フロー

### ステップ1: 本文を取得する

`<SKILL_DIR>/scripts/fetch_article.py` を実行して本文テキストを得る。
`<SKILL_DIR>` は本 SKILL.md のあるディレクトリ（`~/.claude/skills/read-article`）。

```bash
python3 ~/.claude/skills/read-article/scripts/fetch_article.py "<URL>"
```

出力の先頭メタ行を必ず確認すること:

- `TITLE:` … 記事の正式タイトル（要約のタイトルに使う）
- `PAGES:` … **これが出ていたら本文は一部でしかない**。「全文ではない」前提で要約し、
  ユーザーに「複数ページ記事のため1ページ目のみの要約です」と添える。必要なら
  続きページ（`?page=2` 等）を追加取得する。
- `PAYWALL:` … 本文が途中で切れている可能性。要約にその旨を添える。
- `CHARS:` … 本文の文字数。極端に少なければ取得失敗を疑う。

### ステップ2: 取得結果に応じて分岐

- **exit 0**: 本文が取れた。ステップ3へ。
- **exit 2**（本文ほぼ空 / SPA の可能性）: `WebFetch` で同じ URL を
  「本文をそのまま全文抽出して。要約・省略はしないで」と指示してフォールバック取得する。
  それでも取れなければユーザーに「本文を取得できなかった」と報告。
- **exit 1**（取得失敗）: URL・ネットワークを疑い、ユーザーに報告。

> フォールバックで WebFetch を使う場合も、**要約させず本文を出させる**こと。要約は本体が行う。

### ステップ3: Claude 本体が本文を読んで要約する

取得した本文を**自分で読んで**要約する。`fetch_article.py` の BODY と
メタ情報（PAGES/PAYWALL）だけを根拠にし、本文に無い数字・日付・固有名詞を足さない。

ユーザーにはこの形式で提示する:

```
## 【タイトル（TITLE をそのまま or 自然な和訳）】
🔗 <URL>

### 要点
- ポイント1
- ポイント2
- ポイント3

### 詳細メモ
- 補足の事実・数値（本文にあるもののみ）
- 注目すべき主張や結論
```

複数ページ・ペイウォールの注記があれば、要約の冒頭か末尾に明示する。
要約を提示したらこのスキルの責務は完了。記録は行わない。

## エラー時の振る舞い

- **本文取得失敗（exit 1）**: ネットワーク/URL を疑い報告。勝手に推測で要約しない。
- **本文ほぼ空（exit 2）**: WebFetch 全文抽出にフォールバック → それも無理なら報告。
- **複数ページ/ペイウォール検出**: 要約に「一部のみ」と必ず添える。隠して全文のように見せない。

## Out of scope

- 要約結果の記録（デイリーノートへの記録などは行わない。独立スキルとして要約までで完結）
- HN トップ一覧の閲覧（`hn-reader` の担当）
- `?page=` の全ページ自動巡回（現状は検出して知らせるのみ。必要になったら追加する）
- ログインが必要な記事の認証突破
